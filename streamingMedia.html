Windows Streaming Media on IIS
by Richard C. Leinecker MCSD, MCSE

The Internet is moving in a direction that its original designers couldn't have imagined. From a text-based information and communications idiom, it's fast approaching an interactive video entertainment paradigm. The old days of using a text editor to slap together a quick presentation are gone. Users on your Web site expect rich content, especially with the proliferation of broadband technologies such as Cable Modem and DSL.

Important parts of the rich content are audio and video. The technology for delivering this content type has come a long way. Several years ago you might wait ten or fifteen minutes for an AVI file to get to your computer. Then with baited anticipation, you watched a thirty second video clip. I always felt disappointed when my fifteen-minute download resulted in such a miniscule presentation. Now almost everyone with a 56K or better connection can see longer video clips that are virtually uninterrupted.

The technology that's performing this magic falls into three rough categories: well-engineered server services, slick client-side applications, and new compression codecs. This article gives an overview of all three. You'll also learn how to install the server services and use the Windows Media Player in an HTML page to view streamed video.

In future articles I'll show you how to use the Microsoft tools to create, edit, and manipulate content; how to create Visual C++ applications to write content to disk files; and how to create your own Web radio station.

I can hear the pundits from where I sit all the way in Reidsville, North Carolina. They're thinking "I'm making technical presentations that couldn't possibly benefit from video." Who wants to see a developer talk about COM dual interfaces or the COM+ Catalog? But the fact is, we all get more out of a video presentation, especially for technical topics. The best example I can give you is Microsoft's online presentation of COM+ topics. Instead of having a series of articles introducing COM+ topics, they have video presentations of developers and project managers talking about the various topics. When I was learning COM+, I found myself going back many times because I learned from those presentations far more effectively than I learned from reading an article or a chapter in a book.

Simple HTML Video

Strictly speaking, you don't need streaming media to make AVI content accessible from your Web pages. The problem, of course, is that the entire AVI must downloaded before it plays. While this may not be the best option today, it's important to show you how it's done as a basis for techniques I cover later in the article.

The two most common ways to add AVIs to a Web page are to create a link that opens the AVI, and to embed the AVI in the HTML using the EMBED tag. The following code shows a link that opens and AVI:

Opening an AVI file with a link

<A HREF="oxydeoxy.avi">Deoxyribonucleic Acid (4627858 bytes)</A>

You can view this link by clicking here.

The following code shows how to embed an AVI in HTML code:

Embedding an AVI into HTML

<EMBED SRC="oxydeoxy.avi" AUTOSTART= "true" WIDTH="331" HEIGHT= "264" ALIGN="center">

You can view this embedded AVI by clicking here.

Compression Codecs

Uncompressed video takes up huge amounts of space. In order to make desktop video feasible, video compression algorithms were created. Videos are compressed by the removal of redundant audio and video data. For more dramatic size reduction, less important data may also be removed, resulting in some sound and/or image degradation.

A codec (compressor/decompressor) is the algorithm that handles the compression and decompression of your video or audio. Every architecture has certain codecs built in, and some codecs are used in multiple architectures.

Each codec has its strengths and weaknesses. Some are more appropriate for storing video, and some are more appropriate for still images. Some are good for CD-ROM, some for the Web, and yet others for editing videotapes. Some are very slow at compressing but decompress very fast (these are called asymmetrical), and some do both at about the same rate (symmetrical). Some of them compress video to one hundredth of its original size, and at least one of them does no compression at all.

Each codec takes advantage of different properties of an image to achieve their compression, so the type of material being compressed significantly affects not only how much compression can be applied, but also how well the codec can reproduce that compressed material.

The codecs listed below are included in the Windows Media client install. The Windows client is built on top of the DirectShow architecture, and can therefore use its own codecs, and also those for DirectShow, Video for Windows (ACM/VCM). The Macintosh client can use its own codecs, as well as QuickTime codecs. However, to ensure all viewers can see your Windows Media content, it is best to use codecs from the list below.

Advanced Streaming Format

Advanced Streaming Format (ASF) is a file format that stores audio and video information and is specially designed to run over networks like the Internet. It is a highly flexible and compressed format that contains streaming audio, video, slide shows, and synchronized events. ASF enables content to be delivered to you as a continuous flow of data with little wait time before playback begins. This means that you no longer have to wait for your audio and video files to fully download before starting to view them. You simply launch the Windows Media Player and begin.

For example, you can spend more than 40 minutes to download an uncompressed AVI file that's 40 seconds long over a 28.8 Kbps modem, and that requires 1.5 Mbps in bandwidth to play it back. It would be virtually impossible to deliver training presentations that run approximately 20 minutes each segment. When that AVI file is compressed and converted to ASF, it begins playing after only a few seconds. The file can be of virtually unlimited length, and can run over Internet bandwidths.

If you already have audio or video on your site in a popular format, such as WAV or AVI, it's easy to convert it to ASF with Windows Media tools (covered in the next article). If you don't have any digitized media on your site, you can easily record audio or video and convert it to ASF.

Recording audio is easy. Just use a microphone and sound editor, such as Sound Recorder provided with Windows, or another sound editor of your choice. Most laptops have built-in microphones, and popular sound cards have inputs for microphones. Recording video requires a video camera and a capture card.

Once you have an ASF file, hosting on a server is simple. Web publishers who are just getting started with streaming media and who don't expect high volume viewing of the ASF content can simply place the file on any HTTP server, as they would a WAV or AVI file. For experienced Web publishers who require higher quality and better server resource utilization, it makes sense to host the ASF content on a dedicated streaming server.

To post to an HTTP server, simply place both the ASX and ASF files in the same folder as your Web page as described earlier. To post to a Windows NT Services Netshow Server, install Windows NT Netshow Services (covered later in this article) and place the ASF file in an ASF root directory. Place the ASX file in the same folder as the Web page.

In either scenario, for people to be able to view your media, they need to have certain settings called MIME types correctly configured. Installation of the Windows Media Player on most browser configurations usually takes care of these settings. However, for some configurations, you may need to manually set client or server MIME associations.

Installing Media Services

Before your Web server can service the request for streaming media content, you'll have to install the media services. For NT servers you'll need to download the Windows Media Services installation program. For Windows 2000 servers, the services can be installed directly from the Windows 2000 CD.

The installation is straightforward until you get to the dialog asking if you want to enable HTTP streaming. In most cases, the answer will be no. This option is available in case there is a firewall that you'll have to go through. But HTTP streaming is going to cause a conflict if another service (such as IIS) is bound to port 80. As you can see, Microsoft has made streaming media an easy piece to add to any IIS-based Web server. It's a way to bring rich content to users and greatly enhance your presentations.
